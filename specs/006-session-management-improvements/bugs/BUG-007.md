# Bug Report: Running-Session Delete Warning Does Not Appear

**Type**: Implementation drift
**Severity**: High
**Feature**: 006-session-management-improvements
**Reported**: 2026-04-26
**Status**: Patched
**Patched**: 2026-04-26

## Description

When deleting a session that is currently running, the Settings dialog does not show the extra running-session warning. The delete flow proceeds like a normal delete, which means the user does not get the stop-first warning required by the feature.

That removes an important safety step from the destructive flow.

## Root Cause Analysis

The `listen('event.start_session', ...)` handler in `src/renderer/src/App.svelte` was calling `neuzosBridge.sessions.stop(sessionId)` as a pre-start cleanup step. This sent a `session.stop` IPC to the main process, which immediately called `runningSessionIds.delete(sessionId)` — the same session that `session.start` had just added. The result: `runningSessionIds` was always empty at the time any delete dialog opened.

**Fix (T021)**: Replaced `neuzosBridge.sessions.stop(sessionId)` with direct `stopClient()` calls on each layout (same pattern as `event.stop_session`). This resets the UI client without sending IPC, so `runningSessionIds` is preserved after session start.

4. Add a smoke test that deletes a running session and confirms the warning text appears before confirmation.
