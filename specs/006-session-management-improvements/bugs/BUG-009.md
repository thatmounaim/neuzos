# Bug Report: Delete Failure Does Not Show a User Message

**Type**: Implementation drift
**Severity**: High
**Feature**: 006-session-management-improvements
**Reported**: 2026-04-26
**Status**: Patched
**Patched**: 2026-04-26

## Description

When partition deletion fails, the UI does not show the required visible error message. From the user's perspective, delete appears to do something and then silently fails, which makes the flow look successful even though the partition was not removed.

This is a feedback bug as well as a destructive-action bug: the user needs a clear failure dialog when delete cannot complete.

## Root Cause Analysis

The `deleteErrorModal` UI and the renderer error-handling branch were correctly implemented. The error dialog never appeared because `rimraf` was not throwing — it was running against a folder that Electron had recreated (via `clearStorageData()`), and on Windows `rimraf` can succeed or partially succeed without throwing when files are immediately re-locked. The result path was `{ success: true }` despite the folder persisting.

**Fix (T020)**: Removing `clearStorageData()` from the delete handler means `rimraf` now runs against a folder with no active Electron session references. If `rimraf` fails (genuine lock), it throws, the retry loop exhausts, and `{ success: false, error }` is returned — which correctly opens `deleteErrorModal`.

## Recommended Fix

1. Confirm the `session.delete` failure result is being propagated back to the renderer.
2. Verify `deleteErrorModal` is set when the delete IPC returns `{ success: false }`.
3. Add a smoke test that forces partition deletion to fail and checks for the visible error dialog.
4. Make sure the dialog remains visible until the user dismisses it.
