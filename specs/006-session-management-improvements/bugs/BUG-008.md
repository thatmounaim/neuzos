# Bug Report: Deleted Session Leaves Config and Partition State Out of Sync

**Type**: Implementation drift
**Severity**: High
**Feature**: 006-session-management-improvements
**Reported**: 2026-04-26
**Status**: Patched
**Patched**: 2026-04-26

## Description

After deleting a session, the next run does not behave correctly because the app's config state and the real partition folder state are no longer aligned. In practice, the session may look removed in Settings while the underlying partition folder still exists, or the app may try to reopen a session using stale state that no longer matches disk reality.

This is the broader consistency bug behind the delete flow: the config model and the filesystem model are diverging.

## Root Cause Analysis

This is a downstream effect of BUG-006. Because `clearStorageData()` caused the partition folder to survive deletion, the renderer removed the session from config (believing delete succeeded) while the folder remained on disk. On the next launch, no session config entry referenced the stale folder — it just sat there.

**Fix (T020)**: Same as BUG-006 — removing `clearStorageData()` from the delete handler ensures `rimraf` actually removes the folder, keeping config and filesystem in sync.

## Recommended Fix

1. Confirm the delete handler updates or removes the session entry and partition folder as one atomic lifecycle change.
2. Check for stale partition references in the session start path after deletion.
3. Add a restart-focused smoke check: delete a session, restart the app, and verify the deleted session cannot be launched from stale partition data.
4. If the config and filesystem are intentionally separate, add explicit reconciliation logic so deleted partitions are not reused.
