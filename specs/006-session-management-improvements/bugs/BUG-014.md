# Bug Report: Partition Folder Persists on Disk After Successful Session Delete

**Type**: Implementation gap — Electron/Chromium lifecycle
**Severity**: Critical
**Feature**: 006-session-management-improvements
**Reported**: 2026-04-26
**Status**: Patched
**Patched**: 2026-04-26

## Description

After a session is deleted from the neuzOS UI (session disappears from the list, `{ success: true }` is
returned), the session's partition folder at
`%APPDATA%\neuzos\Partitions\persist\<sessionId>\` **remains on disk**. No error dialog is shown,
so from the user's perspective the delete appeared to succeed. This behaviour reproduces for both:

- **Running sessions** (shown stop-warning → stopped → deleted from UI → folder persists)
- **Previously-run sessions that are currently stopped** (deleted from UI → folder persists)

This is distinct from BUG-012 (IPC loop) and BUG-013 (premature ACK), both of which have been
fixed. The folder persistence is now caused by a **Chromium internal recreation race** that is
independent of those fixes.

## Root Cause Analysis

### Why rimraf "succeeds" but the folder comes back

When `session.delete` calls `await rimraf(partitionFolderPath)`:

1. `rimraf` (via Node's `fs.rm({ recursive: true })`) begins deleting files in the partition
   directory.
2. On Windows NTFS, deleting a file that is **open by another process** is allowed — the file is
   unlinked from the directory immediately, but its data remains accessible to the process that
   holds a handle until that handle is closed. So `rimraf` returns **success** even though
   Chromium's handles are still open.
3. However, **LevelDB** (the on-disk storage backing Electron sessions) continuously monitors
   its files. When it detects that its `LOCK`, `MANIFEST`, or `LOG` files have been unlinked, it
   **immediately recreates the directory structure** to maintain database integrity.
4. This recreation races with or follows `rimraf`. After `rimraf` completes, the partition
   directory is back — possibly with new, fresh LevelDB files.

### Why the 2 000 ms grace is insufficient

The `tick()` fix (BUG-013) ensures the `<webview>` DOM element is removed before the ACK is
sent. But **removing the `<webview>` element from the DOM does not synchronously destroy the
Electron WebContents** associated with it. The WebContents destruction (and with it, LevelDB
teardown and handle release) is deferred and can take 2–5+ seconds on Windows:

```
t=0ms     tick() → <webview> removed from DOM
t=~0ms    ACK sent → main starts 2000ms grace
t=~50ms   Electron begins WebContents teardown (async)
t=~500ms  Chromium renderer process starts shutting down
t=~1000ms LevelDB begins flushing and closing files
t=2000ms  Grace expires → rimraf runs ← Chromium STILL active
t=~2500ms Chromium finishes LevelDB teardown (too late)
```

Effectively the grace period starts too soon relative to when Chromium actually releases its
file-system handles.

### Why no error dialog appears

`rimraf` on Windows does **not throw** for unlinked-but-recreated files. The call returns
`true` (success). The outer retry loop catches errors, but since rimraf does not throw, no
retry occurs. The handler returns `{ success: true }`, the renderer removes the session from
the list, and the user sees no error.

    scenarios.

## Recommended Fix

In `src/main/index.ts`, `ipcMain.handle("session.delete")`:

```ts
// 1. Increase grace period so WebContents teardown is more likely complete
await new Promise(resolve => setTimeout(resolve, 5000));

// 2. Use rimraf with internal retries for ENOTEMPTY
// 3. Verify the folder is actually gone after rimraf returns
const maxAttempts = 8;
for (let attempt = 1; attempt <= maxAttempts; attempt++) {
  try {
    await rimraf(partitionFolderPath, { maxRetries: 5, retryDelay: 1000 });
    // Chromium LevelDB may recreate the folder even after rimraf "succeeds" —
    // check that it is truly gone before declaring victory.
    if (fs.existsSync(partitionFolderPath)) {
      throw new Error("Partition folder recreated by Chromium after rimraf");
    }
    deleteResult = { success: true };
    break;
  } catch (err: any) {
    if (attempt < maxAttempts) {
      await new Promise(resolve => setTimeout(resolve, 1200));
    } else {
      deleteResult = { success: false, error: `Could not delete session data: ${err?.message ?? String(err)}` };
    }
  }
}
```
