# Bug Report: Infinite IPC Loop When autoDeleteCache=true Floods Terminal and Recreates Deleted Partition

**Type**: Implementation drift
**Severity**: Critical
**Feature**: 006-session-management-improvements
**Reported**: 2026-04-26
**Status**: Open

## Description

When a session has `autoDeleteCache: true`, stopping the session creates an infinite IPC loop that:

1. Floods the terminal with endless `stop_session <id>` console messages
2. Prevents sessions from staying loaded (the webview is immediately re-stopped)
3. Causes session partition folders to reappear after a successful `rimraf` delete

### Loop Trace

```
stopClient() [started = true]
  → void neuzosBridge.sessions.clearCache(session.id)       [IPC to main]

main: session.clear_cache handler
  → win.webContents.send("event.stop_session", sessionId)   [IPC back to renderer]
  → session.fromPartition(...).clearCache()

renderer: event.stop_session listener
  → neuzClient.stopClient()  [started is now false, but clearCache still fires]
    → void neuzosBridge.sessions.clearCache(session.id)     [IPC to main again]

→ LOOP (continues indefinitely)
```

### How This Breaks Deletion

The `deletingSessionIds` set blocks `session.clear_cache` *during* the `session.delete` handler. However, the loop queues multiple `session.clear_cache` IPC messages before deletion starts. The timeline:

1. `session.delete` runs → `deletingSessionIds.add(id)` → sends `event.stop_session`
2. `stopClient` fires `clearCache` IPC (if autoDeleteCache=true)
3. Main blocks the IPC (`deletingSessionIds.has(id)` = true) ✓
4. ACK received → 2-second grace → `rimraf` succeeds
5. **`deletingSessionIds.delete(id)` runs**
6. Any still-queued `session.clear_cache` messages from the loop now pass the guard
7. `session.fromPartition("persist:" + sessionId).clearCache()` is called
8. Electron's `fromPartition()` **recreates the partition directory**
9. The folder rimraf just deleted is back

### How This Breaks Session Loading

With the loop running, `event.stop_session` fires continuously. Every time the user starts a session (`startClient()` → `started = true`), the loop's next `event.stop_session` immediately calls `stopClient()` again (`started = false`). The webview flashes on screen and disappears — the user sees sessions as "not loading".

  - **Gap**: The spec assumes queued `session.clear_cache` IPCs can be fully suppressed by `deletingSessionIds`. It does not account for IPCs arriving *after* `deletingSessionIds.delete(sessionId)` due to loop-generated backlog.
- **Affected requirement**: FR-003 — the partition folder is being recreated after rimraf, violating the delete guarantee.

- **Impact**: The plan does not describe the `session.clear_cache` handler's behavior when invoked outside the delete context, specifically that it sends `event.stop_session` which creates a reentrance cycle through `stopClient`.

### tasks.md

## Root Cause Analysis

Two cooperating defects:

1. **`session.clear_cache` handler (src/main/index.ts ~L1729)** sends `event.stop_session` back to the renderer before calling `clearCache()`. This is unnecessary — `clearCache()` does not require the webview to be stopped, and sending stop triggers `stopClient` which immediately fires another `clearCache` IPC.

2. **`stopClient` (src/renderer/src/components/Shared/NeuzClient.svelte ~L131)** calls `neuzosBridge.sessions.clearCache()` unconditionally whenever `session.autoDeleteCache` is true, regardless of whether `started` was already `false`. This means every re-entry (caused by the `event.stop_session` from the main handler) fires another `clearCache` IPC.

Neither defect alone creates the loop, but together they form a tight IPC feedback cycle.

### Code fix sketch

**In `NeuzClient.svelte` — guard `stopClient` against re-entry:**
```ts
export const stopClient = (onStopped?: () => void) => {
  if (!started) {
    onStopped?.()   // still fire callback (needed for ACK in delete flow)
    return
  }
  clearSessionHealth()
  started = false
  if (session.autoDeleteCache) {
    void neuzosBridge.sessions.clearCache(session.id)
  }
  onUpdate(session.id)
  koreanLinkFixed = false
  window.electron.ipcRenderer.send('webview.unregister_mouse', { sessionId: session.id })
  onStopped?.()
}
```

**In `src/main/index.ts` — remove `event.stop_session` from `session.clear_cache` handler:**
```ts
ipcMain.on("session.clear_cache", async function (event, sessionId: string) {
  if (deletingSessionIds.has(sessionId)) return;
  // REMOVE: win?.webContents.send("event.stop_session", sessionId);
  const sess = session.fromPartition("persist:" + sessionId);
  await sess.clearCache();
});
```
