# Bug Report: Delete ACK Sent Before Webview Is Removed from DOM

**Type**: Spec gap / Implementation drift
**Severity**: High
**Feature**: 006-session-management-improvements
**Reported**: 2026-04-26
**Status**: Patched
**Patched**: 2026-04-26

## Description

The `event.stop_session_ack` is sent to the main process *synchronously* inside `stopClient()`, immediately after setting `started = false`. However, Svelte's `{#if started}` reactive block does **not** remove the `<webview>` element from the DOM synchronously — the DOM update is batched and runs asynchronously in the next microtask flush (`tick()`).

This means the 2-second grace period in `session.delete` starts while the `<webview>` element is still in the DOM, and Chromium's renderer process still holds file-system handles on the partition directory. On Windows, Chromium's LevelDB compaction tasks can hold these handles for several seconds after the webview is removed from DOM, so even the full 2-second grace may not be sufficient.

### Timing diagram

```
t=0ms      stopClient() called
           started = false         (Svelte schedules DOM update — NOT yet executed)
           onStopped?.()  →  ACK sent to main

t=~0ms     Main receives ACK, starts 2-second grace timer

t=~1ms     Svelte microtask runs, <webview> removed from DOM
           Electron begins destroying webContents (async)

t=~50ms    Chromium renderer process starts LevelDB/SQLite teardown

t=2000ms   Main's grace expires → rimraf() attempted
           ↳ On a slow/busy Windows system Chromium may still hold handles
             → EPERM / EBUSY → rimraf retries (up to 5×, 800ms apart)
             → After all retries, partition folder may remain
```

### Relation to BUG-012

BUG-012 (infinite loop) is the primary cause of deletion failures in the current build. Even after BUG-012 is fixed, this timing issue is an independent, lower-level defect that could still cause sporadic failures on slower Windows machines or when partitions are large.

## Artifact Traceability

### spec.md

- **Affected requirement**: FR-003 — "System MUST … await an explicit renderer acknowledgement that `stopClient()` finished **and the webview was torn down**"
  - **Gap**: The phrase "webview was torn down" is not guaranteed by the current implementation. The ACK is sent when `stopClient()` has *started* its cleanup, not when the `<webview>` element has actually been removed from the DOM.

### plan.md

- **Affected sections**: BUG-011 plan note (plan.md ~L195) — describes the ACK + grace approach but does not specify that the ACK must wait for Svelte's DOM flush before being emitted.
- **Impact**: The plan should be updated to note that the ACK sender must wait for the Svelte tick (DOM update) before firing, so the grace period starts only after the webview element is confirmed removed.

### tasks.md

- **Affected tasks**:
  - T024 [BUG-011] — "Add `event.stop_session_ack` IPC channel …" is done, but the ACK is sent too early (before DOM flush)
  - T027 — manual Windows smoke test; this defect is exactly what T027 is designed to catch
- **Missing tasks**: A new task is needed to use Svelte's `tick()` in `stopClient` (or in App.svelte's `event.stop_session` handler) to ensure the DOM update is processed before the ACK is sent.

## Root Cause Analysis

`stopClient` in `NeuzClient.svelte` performs:
```ts
started = false        // Svelte state mutation — DOM update deferred
...
onStopped?.()          // ACK fired synchronously — webview still in DOM
```

Svelte 5 rune-based state changes (`$state`) are batched similarly to Svelte 4 — the DOM is updated in the next microtask. `onStopped?.()` is called in the same synchronous call stack as `started = false`, so it fires before the `<webview>` element is removed.

The spec requirement for "webview torn down" is therefore never actually met before the ACK.

### Code fix sketch

**Option A — use `tick()` in `stopClient` (preferred):**
```ts
import { tick } from 'svelte'

export const stopClient = (onStopped?: () => void) => {
  if (!started) {
    onStopped?.()
    return
  }
  clearSessionHealth()
  started = false
  if (session.autoDeleteCache) {
    void neuzosBridge.sessions.clearCache(session.id)
  }
  onUpdate(session.id)
  koreanLinkFixed = false
  window.electron.ipcRenderer.send('webview.unregister_mouse', { sessionId: session.id })
  // Wait for Svelte to flush the DOM update (removes <webview>) before firing ACK
  if (onStopped) {
    void tick().then(onStopped)
  }
}
```

**Spec clarification needed**: FR-003 should explicitly state that the ACK MUST NOT be sent until Svelte's DOM flush completes (i.e. the `<webview>` element is removed from the DOM). The current wording "webview was torn down" is ambiguous.
