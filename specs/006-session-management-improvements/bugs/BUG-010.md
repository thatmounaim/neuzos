# Bug Report: Partition Folder Recreated After Delete Due to autoDeleteCache Race Condition

**Type**: Implementation drift
**Severity**: High
**Feature**: 006-session-management-improvements
**Reported**: 2026-04-26
**Status**: Patched
**Patched**: 2026-04-26

## Description

After a confirmed session delete, the partition folder on Windows is still present on disk even though the UI reports success. The partition deletion runs correctly, but the folder reappears immediately afterward because the `session.clear_cache` IPC handler recreates it.

**Race condition sequence**:
1. `session.delete` handler sets `event.stop_session` to renderer.
2. Renderer's `stopClient()` fires. If that session has `autoDeleteCache = true`, `stopClient()` calls `neuzosBridge.sessions.clearCache(sessionId)`, which sends `session.clear_cache` back to main.
3. Main waits 5 seconds (stop-wait), then rimraf deletes the partition folder. ✓
4. `session.clear_cache` handler (received earlier, queued on the event loop) then calls `session.fromPartition("persist:" + sessionId).clearCache()`.
5. Electron's `session.fromPartition()` call recreates the partition folder. ✗

Even sessions with `autoDeleteCache = false` are affected in edge cases where clear_cache is triggered from elsewhere.

## Root Cause Analysis

The `session.clear_cache` IPC handler had no awareness of an in-progress deletion. It unconditionally accessed `session.fromPartition()`, which on Windows causes Electron's Chromium layer to recreate the `Partitions/persist/<id>/` directory. Rimraf completes successfully but the folder reappears in the same event loop tick.

**Fix**: Added `deletingSessionIds: Set<string>` in `src/main/index.ts`.  
- `session.delete` handler calls `deletingSessionIds.add(sessionId)` **before** sending `event.stop_session`.  
- `session.clear_cache` handler returns early if `deletingSessionIds.has(sessionId)`.  
- `session.delete` handler calls `deletingSessionIds.delete(sessionId)` after rimraf completes (success or final failure).

## Recommended Fix

Already applied:
1. `deletingSessionIds` Set added at module scope alongside `runningSessionIds`.
2. `session.delete` adds ID to set before stop, removes after rimraf.
3. `session.clear_cache` skips `fromPartition()` call if ID is in `deletingSessionIds`.
