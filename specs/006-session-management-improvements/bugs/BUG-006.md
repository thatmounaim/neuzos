# Bug Report: Session Delete Leaves Partition Folder Behind

**Type**: Implementation drift
**Severity**: High
**Feature**: 006-session-management-improvements
**Reported**: 2026-04-26
**Status**: Patched
**Patched**: 2026-04-26

## Description

Deleting a session from Settings appears to succeed after the normal stop-and-wait delay, but the session's partition folder is still present on disk afterward. That means the UI/config state says the session is gone, while the actual partition data was not removed.

This is a data-lifecycle bug because the delete flow is only partially completing the destructive operation.

## Root Cause Analysis

`ipcMain.handle("session.delete")` in `src/main/index.ts` was calling `session.fromPartition("persist:" + sessionId).clearStorageData()` after the 5-second stop-wait. Accessing the Electron session object causes Chromium to re-open or recreate the partition directory on Windows before `rimraf` runs. `rimraf` then either finds locked files or deletes a folder that reappears immediately, returning `{ success: true }` while the folder persists.

**Fix (T020)**: Removed the `clearStorageData()` block entirely. The `rimraf` call alone is sufficient — it removes all on-disk data including what `clearStorageData` would have cleared, and it now runs against a folder that Electron is no longer touching.

## Recommended Fix

1. Re-check the `session.delete` handler in `src/main/index.ts` to ensure the resolved partition path points at the real on-disk folder.
2. Confirm the delete routine is not only clearing storage but also removing the partition directory itself.
3. Add a runtime smoke check for the deleted partition folder path after confirmation succeeds.
4. Re-run the deletion flow on Windows to verify the folder is gone and the session cannot be restarted from stale partition data.
