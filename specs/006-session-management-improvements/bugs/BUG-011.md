# Bug Report: Partition Folder Still Not Deleted — Fire-and-Forget Stop Has No ACK

**Type**: Implementation gap
**Severity**: High
**Feature**: 006-session-management-improvements
**Reported**: 2026-04-26
**Status**: Patched
**Patched**: 2026-04-26

## Description

After a confirmed session delete, the partition folder on Windows is **still present on disk** despite
all BUG-010 guards being in place. The `deletingSessionIds` guard correctly blocks
`session.clear_cache` IPC messages from recreating the folder, but the folder either:

1. Is never fully deleted by rimraf because Windows file handles are still open at the time rimraf
   runs (the 5-second blind wait is insufficient), **or**
2. Is deleted by rimraf but then partially recreated by Electron's asynchronous data-flush
   (SQLite WAL checkpoint, IndexedDB compaction) which continues after the webview WebContents is
   destroyed and outlasts the 5-second grace period.

The user never sees an error dialog, which means rimraf is either resolving without throwing (force
mode skipping locked entries silently) or the folder reappears after a brief absence.

## Root Cause Analysis

### The fire-and-forget stop model

The `session.delete` handler sends `event.stop_session` to the renderer and then **blindly waits
5 seconds** with no acknowledgement from the renderer confirming that:

- `stopClient()` has run and set `started = false`,
- Svelte's DOM reconciliation has actually removed the `<webview>` element,
- Electron has destroyed the associated WebContents,
- All async data flushes (SQLite WAL, IndexedDB, cookie writes) have completed and file handles are
  closed.

On Windows, all four steps are asynchronous and their combined latency is non-deterministic. Under
load, after a long session, or on slower hardware, this can easily exceed 5 seconds.

### Chromium async flush after WebContents destruction

When the `<webview>` element is removed from the DOM, Electron schedules destruction of the
underlying WebContents. Before the WebContents is fully torn down, Chromium checkpoints any open
SQLite databases (Cookies, IndexedDB) and flushes IndexedDB write queues. These operations run on
Chromium's I/O thread and may complete **after** the main process has already called rimraf. The
net effect on Windows: rimraf succeeds, then Chromium's flush completes and recreates
`Cookies-journal`, `IndexedDB/`, or similar files inside the (now-absent) partition folder,
re-creating the directory in the process.

### Secondary issue — `session.clear_storage` wrong path and missing guard

The `session.clear_storage` IPC handler (separate from deletion) builds the rimraf path as:
```
path.resolve(partitionsBase, sessionId)   // ← missing 'persist' subfolder
```
instead of:
```
path.resolve(partitionsBase, 'persist', sessionId)
```
This means its rimraf call targets a non-existent path and silently does nothing. Additionally,
this handler has no `deletingSessionIds` guard and calls `session.fromPartition().clearStorageData()`
unconditionally, which could interfere if it were ever called during deletion. This is tracked here
as a secondary defect to address in the same fix pass.

## Proposed Fix

### 1. Implement an ACK round-trip for the stop-before-delete flow

Replace the blind 5-second wait with a proper acknowledgement:

**Renderer side** (`App.svelte` or `NeuzClient.svelte`):
When an `event.stop_session` event is received as part of a delete (indicated by an optional
`{ requestAck: true }` payload), call `stopClient()`, then await Svelte's next DOM reconciliation
tick, then send `event.stop_session_ack` back to the main process via IPC.

**Main process** (`session.delete` handler):
After sending `event.stop_session`, `await` a Promise that resolves when
`event.stop_session_ack` is received from the renderer, with a 12-second timeout fallback so the
delete never hangs.

After receiving the ACK (or timeout), add an **additional 2-second grace period** to allow
Chromium's I/O thread to finish in-flight flushes before rimraf runs.

```ts
// Conceptual shape — exact implementation in tasks
mainWindow?.webContents.send("event.stop_session", sessionId, { requestAck: true });
await Promise.race([
  new Promise<void>(resolve =>
    ipcMain.once("event.stop_session_ack", (_e, id) => { if (id === sessionId) resolve(); })
  ),
  new Promise<void>(resolve => setTimeout(resolve, 12_000)) // timeout fallback
]);
await new Promise(resolve => setTimeout(resolve, 2_000)); // Chromium I/O flush grace
// → rimraf
```

### 2. Fix `session.clear_storage` path and add guard

- Change `path.resolve(partitionsBase, sessionId)` → `path.resolve(partitionsBase, 'persist', sessionId)`.
- Add `if (deletingSessionIds.has(sessionId)) return;` at the top of the handler.
