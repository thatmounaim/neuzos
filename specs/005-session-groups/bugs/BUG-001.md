# Bug Report: Session Delete Does Not Persist to Config

**Type**: Implementation drift  
**Severity**: Critical  
**Feature**: 005-session-groups  
**Reported**: 2026-04-25  
**Status**: Patched  
**Patched**: 2026-04-25

## Description

Deleting a session via the Delete button in Settings > Sessions removes the partition folder on disk (the main-process IPC handler handles that correctly), but the renderer never calls `config.save` afterwards. As a result:

1. The session **disappears from the UI immediately** (in-memory state is updated).
2. The **config file on disk still contains the deleted session entry**.
3. On **application restart**, the config is re-loaded from disk and the deleted session reappears in the session list — even though its partition data is gone.
4. The session also **remains visible in the Settings tab on any re-render that forces a config reload** (e.g. config import, export round-trip).

Additionally, the group cleanup that runs after delete (`ensureSessionGroups().forEach(...)`) also operates only on the in-memory `neuzosConfig` state and is never flushed to disk, so orphaned `sessionIds` survive in the saved `sessionGroups` array.

## Artifact Traceability

### spec.md

- **Affected edge case**: "What happens if all sessions are deleted while groups remain? Groups persist (empty) and are shown with no session rows inside them." — This implies deletion is permanent and the config is updated.
- **Affected edge case**: "What happens when the only session in a group is deleted? The group remains but is empty." — Requires the session id to actually be removed from `sessionIds` in the persisted config.
- **Affected requirement**: FR-001 — `sessionGroups[].sessionIds` must not contain stale references to deleted sessions after a save cycle.
- **Gap identified**: The spec does not explicitly state "a config save must follow session delete", but it is implicitly required for the delete to be permanent. The spec should be amended to add this explicit contract.

### plan.md

- **Affected sections**: "Storage" — "written via `saveConfig()` in main process after every `config.save` IPC call". The IPC `session.delete` handler never triggers `config.save`; the renderer must issue a separate `config.save` call after updating the in-memory config.

### tasks.md

- **Affected tasks**:
  - **T004** `[X]` — "including removing deleted session ids from every group's `sessionIds` list when a session is deleted" — marked complete, but the fix only runs in memory; the result is never persisted.
  - **T009** `[X]` — "Keep the saved `sessionGroups` order stable by writing reorder and delete mutations back through config save" — The group delete path calls `saveConfig` via config.save, but the **session delete path** does not.
- **False completions**: T004 is functionally incomplete because the in-memory mutation is never flushed to disk.
- **Missing tasks**: A new sub-step inside T004 is needed: after successfully deleting the session, call `neuzosBridge.config.save(neuzosConfig)` (or an equivalent IPC call) so both `sessions` and `sessionGroups` are persisted atomically.

## Root Cause Analysis

The `deleteSession` function in `SessionSettings.svelte` (line ~280) calls `neuzosBridge.sessions.deleteSession(sessionId)` and then updates `neuzosConfig.sessions` and `neuzosConfig.sessionGroups` in memory, but it **never calls a config save IPC handler**. Compare this to other mutation paths (e.g., reorder, group add/delete) which persist via the save button or reactive config saves. Session deletion skips this step entirely.

## Recommended Fix

1. Run `/speckit.bugfix.patch` to:
   - Amend spec.md edge case section to explicitly state "session deletion persists the updated session list and group membership to disk".
   - Amend T004 to add a sub-requirement: "call config save IPC after in-memory state is updated".
2. In `SessionSettings.svelte`, at the end of the `if (result.success)` block inside `deleteSession`, add a call to `neuzosBridge.config.save(neuzosConfig)` (serialised) so the updated `sessions` and `sessionGroups` arrays are written to disk.
3. Run `/speckit.bugfix.verify` to confirm consistency.
4. Resume `/speckit.implement` for the code fix.
