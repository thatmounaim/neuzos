# Bug Report: Session Delete Fails — Wrong Partition Path + Insufficient Stop Wait

**Type**: Implementation drift  
**Severity**: Critical  
**Feature**: 005-session-groups (cross-cutting: session management)  
**Reported**: 2026-04-25  
**Status**: Patched  
**Patched**: 2026-04-25

## Description

Deleting a session still does not work reliably. The session remains visible in Settings after the delete attempt and the partition folder on disk is never removed. Two compounding root causes were identified in the main-process `session.delete` IPC handler:

1. **Wrong partition path**: The handler constructs `<userData>/Partitions/<sessionId>` but Electron stores `persist:<sessionId>` partition data at `<userData>/Partitions/persist/<sessionId>` (note the extra `persist/` subdirectory). `rimraf` targets a non-existent path, succeeds silently, and the actual folder is never touched.

2. **Fire-and-forget stop + fixed wait**: The handler sends `event.stop_session` to the main window (correct target) but does not wait for an acknowledgment that the webview has actually closed before starting the 2500 ms countdown. On Windows, an actively-loaded Chromium webview can hold file handles for several seconds after the stop signal. If the handles are not released within the 2500 ms window, all five rimraf retry attempts (5 × 800 ms = 4 s) also fail, and the IPC returns `{ success: false }` — which surfaces an error dialog. The user dismisses the dialog, the session is still in the list, and the folder persists.

The BUG-001 fix (config.save after delete) is logically correct but never runs because `result.success` is almost always `false` under these two failure modes.

## Recommended Fix

1. Change the partition path construction:
   ```diff
   - const partitionFolderPath = path.resolve(partitionsBase, sessionId);
   + const partitionFolderPath = path.resolve(partitionsBase, 'persist', sessionId);
   ```
2. Either increase `await new Promise(resolve => setTimeout(resolve, 2500))` to at minimum 5000 ms, or implement a handshake: send `event.stop_session` and wait for a `session.stopped` acknowledgment IPC message from the renderer before proceeding with folder deletion.
3. Run `/speckit.bugfix.verify` then resume `/speckit.implement`.
