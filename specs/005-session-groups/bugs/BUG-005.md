# Bug Report: Mouse Button Keybinds (Middle, Mouse4, Mouse5) Do Not Fire While Gaming in Webview

**Type**: Spec gap / Implementation drift  
**Severity**: High  
**Feature**: 005-session-groups (cross-cutting: keybind system)  
**Reported**: 2026-04-25  
**Status**: Patched  
**Patched**: 2026-04-25

## Description

Keyboard keybinds work correctly while the game is running in the session webview. Mouse button binds (Middle, Mouse4, Mouse5) can be successfully **recorded** in the KeyBinder UI and are correctly **stored** in the config. However, they **never fire** while the user is actively gaming (focus is inside the game's webview).

The binds work only when the user clicks a mouse button while focus is outside the webview (e.g., clicking in the Neuzos titlebar or sidebar). As soon as the game webview has focus, mouse button events are silently swallowed.

## Root Cause Analysis

The two keybind delivery paths diverge based on input type:

| Input type | Dispatch path | Works in webview? |
|-----------|---------------|-------------------|
| Keyboard keys | `globalShortcut.register()` in main process → `event.ui_action_fired` → renderer | ✅ Yes — OS-level, bypasses webview isolation |
| Mouse buttons (Middle, Mouse4, Mouse5) | `window.addEventListener('mousedown')` in renderer → `dispatchRendererBind()` | ❌ No — renderer DOM events do not receive events originating inside a `<webview>` |
| Gamepad buttons | `requestAnimationFrame` poll in renderer via `navigator.getGamepads()` | ✅ Works (gamepad API is not isolated per-webview) |

`registerKeybinds()` in `src/main/index.ts` explicitly **skips** mouse keys when registering global shortcuts:
```javascript
if (normalizedKey.startsWith('mouse') || normalizedKey.startsWith('gamepad')) {
  return; // not registered via globalShortcut
}
```

Mouse binds are instead handled by `startMouseListener()` in `uiActionContext.svelte.ts`, which attaches a `mousedown` listener to `window` in the renderer process. In Electron, a `<webview>` tag runs in an **isolated renderer process** (separate Chromium content process). Mouse events inside the webview are processed entirely within that isolated process and are **never propagated** to the parent renderer's `window`. Therefore, Mouse4/Mouse5 clicks inside the game are invisible to the listener.

## Implementation Options

### Option A — `before-input-event` on webview webContents (Recommended)

Each Electron `<webview>` exposes a `webContents` reference accessible via `webview.getWebContentsId()` and `webContents.fromId()` in the main process. The main process can subscribe to `webContents.on('before-input-event', ...)` for each active session webview. Mouse button events appear here as `input.type === 'mouseDown'` with `input.button` identifying the button. The main process can then dispatch the keybind action via `mainWindow.webContents.send('event.ui_action_fired', ...)`.

**Pros**: Pure Electron API, no native addons, no external dependencies.  
**Cons**: Requires tracking webContents IDs per active session; must subscribe/unsubscribe when sessions start/stop.

### Option B — Native global mouse hook (`uiohook-napi`)

A native Node.js addon that installs an OS-level mouse hook (Windows `SetWindowsHookEx`, macOS `CGEvent`, Linux `XRecord`). Receives all mouse events regardless of application focus or webview isolation.

**Pros**: Completely reliable; same pattern as globalShortcut for keyboards.  
**Cons**: Adds a native dependency; requires rebuild per Electron version; higher maintenance burden; potential anti-cheat false-positive risk.

### Recommended path

Implement Option A as the minimal fix. If `before-input-event` coverage proves insufficient (e.g., edge cases with webview focus states), revisit Option B.

## Recommended Fix Steps

1. In the main process (`src/main/index.ts`), when a session webview starts (session IPC receives the webContentsId from the renderer), subscribe to `before-input-event` for that webContents.
2. On `mouseDown` events for buttons 1 (middle), 3 (Mouse4), 4 (Mouse5): look up matching binds in `neuzosConfig.keyBinds` and the active profile, and call `dispatchKeybindEvent(bind)`.
3. On session stop/close, remove the `before-input-event` listener.
4. Remove the renderer-side `startMouseListener` fallback (or keep it as a secondary path for non-webview clicks).
